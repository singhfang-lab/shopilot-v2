# 低分回答回炉规则

> 用途：当 AI 回答不合格时，知道应该改哪一层，而不是盲目重写全部内容。

## 1. 低分原因分类

| 低分原因 | 可能需要修改的层级 |
|---|---|
| 题型识别错误 | 问题意图识别与调用规则 |
| 回答跑偏 | 主卡/辅助卡组合规则 |
| 无数据时乱下结论 | 全局回答协议、禁止未经验证假设规则 |
| 有数据时仍然泛泛而谈 | 该问题卡的“有数据时必须分析”与输出契约 |
| To-do 太虚 | To-do颗粒度规则、该问题卡动作示例 |
| 没有验证标准 | 验证标准规则、该问题卡验证模板 |
| 行业不适配 | 行业场景专题 |
| 语言不自然 | 商户端回答蓝图、禁止表达 |
| 数据定义混乱 | 经营指标数据字典 |

---

## 2. 回炉流程

```text
1. 记录商户原问题
2. 记录 AI 原回答
3. 按评分表打分
4. 标注低分维度
5. 判断是调用问题、规则问题、问题卡问题还是输出示例问题
6. 修改对应文件
7. 用同一个问题复测
8. 复测通过后再推广到同类问题卡
```

## 4. 常见回炉场景

### 场景一：模型使用高级词但不落地

表现：

- 护城河；
- 品牌格调；
- 圈层裂变；
- 体验升级。

处理：

- 在禁止表达中加入该类词；
- 在问题卡中补“必须落到具体动作”；
- 在合格示例中给具体 To-do。

---

### 场景二：模型做未经验证假设

表现：

- 直接说某商圈核心客群是商务宴请；
- 直接说某城市顾客重视送礼；
- 直接说不能靠卖更多产品增长。

处理：

- 加入禁止未经验证假设规则；
- 改成条件表达；
- 引导商户提供对应数据。

---

### 场景三：有数据但没有诊断

表现：

- 复述数据；
- 说“订单下降明显”；
- 但没有原因优先级和具体动作。

处理：

- 在问题卡中补“有数据时必须分析”；
- 强制命中因素、异常定位、原因优先级；
- 补有数据版合格示例。

---

## 5. 不建议的回炉方式

不要只做：

- 增加更多文字；
- 再写一篇类似文章；
- 加更多泛建议；
- 全部重写但不改规则。

真正要改的是：

- 调用规则；
- 输出契约；
- To-do 粒度；
- 验证标准；
- 合格样例。
